1Overview

  • Product: ClaudeClaw — personal AI assistant controlled from Telegram/Discord/iMessage.
  • Core behavior: bridges chat messages to a local Claude Code CLI subprocess using the Claude Code SDK.
  • Key behaviors: session resumption, typing indicator, media handling, optional voice.
  • Extensible via modular Power Packs: Memory v2, Multi-Agent, War Room, Mission Control, Security, Dashboard, Meeting Bot, WhatsApp.
  • Primary risks: Windows service workflow, dependency completeness, and path-handling correctness.

Review before implementation to avoid overbuilding or building the wrong thing.

4Tasks and subtasks

4.1 Bootstrap and repo structure

  • Create project files: .gitignore, package.json, tsconfig.json.
  • Run npm install for selected feature set.
  • Create runtime directories: store/, workspace/uploads/.
  • Validate local claude CLI availability and authentication.

4.2 Core runtime

  • Implement src/env.ts robust parser using only fileURLToPath for path resolution.
  • Implement src/logger.ts with pino plus optional pino-pretty transport in development.
  • Implement src/config.ts exporting named constants (36-46+ env exports) and runtime constants.
  • Implement src/db.ts with WAL mode, inline migrations using PRAGMA table_info() checks.
  • Implement src/agent.ts with query() wrapper, typing callback, session extraction, token tracking for cost footer.
  • Implement src/errors.ts, src/state.ts, src/message-queue.ts, src/cost-footer.ts.

4.3 Platform integration

  • grammy Telegram bot implementation with text, voice, document, photo, video handlers.
  • Markdown-to-Telegram-HTML formatter.
  • Discord or iMessage variant if selected.
  • Auth flow: /chatid discovery, allowlist enforcement.

4.4 Voice

  • Speech-to-text via Groq Whisper/OpenAI Whisper.
  • Text-to-speech via ElevenLabs with fallback chain (Kokoro, macOS say).
  • File handling: rename .oga to .ogg for Groq.
  • Per-chat voice toggle with force voice reply behavior.

4.5 Memory

  • simple: last N turns in SQLite.
  • full_v2: extraction via Gemini, embeddings via Gemini, duplicate detection, consolidation, 5-layer retrieval, decay, nudging, relevance feedback.
  • Adjust implementation order from db.ts schema to memory.ts and ingestion/consolidation modules.

4.6 Multi-agent

  • Agent registry and per-agent session isolation via composite primary key.
  • Orchestrator with @agentId: parsing and delegation hooks.
  • Agent lifecycle: activate, deactivate, restart, delete.
  • Hive mind and inter-agent task tables.

4.7 War Room

  • Pipecat server on port 7860 with live/legacy modes.
  • GoT personas, routing rules, agent bridge in Python and Node.
  • Browser UI bundle and embedded HTML refs.

4.8 Security

  • PIN lock + idle auto-lock + kill phrase.
  • Exfiltration guard with regex patterns for API keys, base64, URL-encoded secrets.
  • Audit logging and blocked-event tracking.

4.9 Dashboard

  • Hono API server on port 3141 with token auth.
  • Embedded HTML dashboard for memory, agents, missions, hive mind, war room management.

4.10 Mission control and scheduler

  • Cron scheduler with stuck-task recovery and 60s polling.
  • Priority async mission queue with optional Gemini auto-assignment.

4.11 WhatsApp bridge

  • whatsapp-web.js client with outbox queue and QR onboarding.
  • Separate daemon process lifecycle and signal handling.

4.12 Entrypoint and lifecycle

  • Startup sequence, PID lock, graceful shutdown, background service integration.
  • Windows lifecycle guidance with PM2 instructions.

4.13 Setup and operations

  • Interactive wizard for API key collection, service installation, chat-ID discovery.
  • Status script and CLAUDE.md templating.

4.14 Verification

  • Typecheck, build, unit/integration tests.
  • Manual smoke test checklist for each enabled feature.
